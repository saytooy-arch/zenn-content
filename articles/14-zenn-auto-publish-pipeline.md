## TL;DR

Zennに技術記事を書き溜めていたら投稿制限に引っかかった。解決策として、claude -pで全記事を俯瞰ブラッシュアップ→1日1件だけ自動公開するパイプラインをcronで構築した。

## きっかけ: 投稿制限に引っかかった

Claude Codeを使った開発で得た知見を技術記事にしていた。下書きが10件以上溜まり、まとめて公開しようとしたところ、Zennの投稿制限が発動した。

短時間に複数のgit pushを行うと制限がかかる仕組みだった。

## Before: 手動で1件ずつ公開

```
記事を書く → 手動でpublished: trueに変更 → git push → 次の記事...
```

問題点:
- 複数記事を一度に公開できない
- 記事間の重複や整合性を個別にしか確認できない
- 投稿制限のタイミングが読めない

## After: 自動パイプライン

```
[毎日12:02 cron]
  → brushup_articles.sh（claude -pで全記事俯瞰ブラッシュアップ）
  → auto_publish.sh（1件だけpublished: true → push）
```

### ブラッシュアップ: 全記事を俯瞰してレビュー

ポイントは「1記事ずつレビュー」ではなく「書き溜めた全記事を一括で俯瞰する」こと。

claude -pに`published: false`の全記事を渡し、以下の7基準でレビューさせる:

1. **単体で読める** — シリーズの前提知識がなくても理解できるか
2. **1記事1テーマ** — 複数テーマが混在していれば分割
3. **エピソード先行** — 抽象論ではなく具体的なエピソードから始まるか
4. **固有名詞を外す** — 自社プロジェクト名を汎用的な表現に置換
5. **Before/After** — 変更の前後が明確か
6. **数字** — 定量的な成果・効果が含まれるか
7. **タイトル** — クリックしたくなるタイトルか

全記事を俯瞰することで、重複テーマの統合や記事の分割判断ができる。個別レビューでは見えない全体最適が可能になる。

### 自動公開: 1日1件だけ

`auto_publish.sh`は単純なロジック:

1. `published: false`の記事を1件探す
2. `published: true`に書き換える
3. git commit & push

1日1件に制限することで投稿制限を確実に回避する。

### レポート出力: 非同期報告

ブラッシュアップ時に統合・分割・大幅修正などの報告事項があれば`reports/`にファイル出力する。次回の対話で確認できる。

## 構成

```
zenn-content/
  articles/          # Zenn記事（published: true/false）
  scripts/
    auto_publish.sh   # 1日1件自動公開
    brushup_articles.sh  # claude -pで全記事ブラッシュアップ
  reports/            # ブラッシュアップ報告
```

crontab:
```
2 12 * * * /path/to/auto_publish.sh
```

## 3つの学び

1. **制約を仕組みで解決する**: 投稿制限という制約に対し、手動ワークアラウンドではなくcron+スクリプトで恒久対策にした
2. **個別最適より全体最適**: 1記事ずつのレビューではなく、全記事を俯瞰することで重複排除・統合・分割の判断ができる
3. **ブラッシュアップを工程に組み込む**: 「公開前に毎回実行」をcronで強制することで、品質が持続的に改善される

## まとめ

投稿制限をきっかけに、結果的により良い運用フローが構築できた。制約は仕組み化のチャンスになる。
