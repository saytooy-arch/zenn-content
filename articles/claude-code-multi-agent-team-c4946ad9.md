個人開発のAndroidアプリを、Claude Codeのエージェント機能で「開発チーム」を組んで作ってみた。結果、障害が100件以上発生し、エージェントが勝手に工程をスキップし、品質管理の仕組みを一から作り直すことになった。

この記事は「AIエージェントを開発に使ってみたい」人に向けて、**1週間の試行錯誤で学んだ「エージェントに仕事を任せる設計パターン」**をまとめたものだ。

## 前提: Claude Codeのエージェント機能

Claude Codeには、エージェントを定義して専門的なタスクを委譲する機能がある。`.claude/agents/` にMarkdownファイルを置くと、それぞれが独立したエージェントとして動作し、使えるツール（ファイル編集、コマンド実行等）やモデル（sonnet/haiku）を制限できる。

この機能を使って、1人の開発者（自分）の代わりに「開発チーム」を組んでみた。

## 作ったもの

**Recipe Scaler** — レシピサイト（クックパッド、クラシル等）を開くと、人数分に食材の分量を自動計算してくれるAndroidアプリ。複数レシピの食材を合算して買い物リストも作れる。

技術的には、Webサイト上のDOMをJavaScriptで操作して分量を書き換える仕組みで、Chrome拡張と同じ発想だ。これをAndroidのWebView上で動かしている。

## Day 1: 動いた

ソースコードはあったが、一度も動作確認していなかった。Claude Codeに「動くようにして」と頼んだら、依存関係を解決し、テスト用のHTMLを作り、ブラウザで動作確認できる状態にしてくれた。

「おお、動いた」。ここまでは順調だった。

ただし1つ問題があった。クラシルのレシピページで食材が認識されない。原因を調べると、CSSセレクタ（ページ上の要素を特定するためのパターン）が `.recipt-ingredient-list__item` になっていた。"recipe" のスペルミスだ。しかも実際のクラス名はまったく別の形式だった。

**AIは実際のサイトを見ずに、クラス名を「推測」してコードを書いていた。**

これが障害台帳の1件目（INC-001）になった。

## Day 2〜3: 障害が止まらない

INC-001を修正したら次の問題が見つかった。修正するたびに新しい障害が発覚する。

| 障害番号 | 内容 | 重要度 |
|---------|------|--------|
| INC-001 | CSSセレクタの推測タイポ | 高 |
| INC-002 | 画面遷移後にUIが更新されない | 高 |
| INC-003 | 同じ要素が二重に処理されて計算結果が倍になる | 高 |
| INC-004 | 削除済みDOM要素へのアクセスでクラッシュ | 中 |
| … | （個別のバグが続く） | |
| INC-027 | Android環境で設定が丸ごと上書きされアプリ起動不可 | **致命的** |
| INC-028 | フレームワークのAPI誤用でデータ連携が全壊 | **致命的** |
| INC-029 | データの型が途中で変わり集計結果がゼロに | 高 |

2日間で30件を超えた。番号が増えるにつれて、障害の性質が変わっていくのがわかる。**INC-001〜004は「コードの間違い」だが、INC-027〜029は「設計の間違い」だ。** 個々のバグを潰しても、次の設計ミスが待っている。

問題の本質はコードではなかった。**開発プロセスが存在しなかった。** 誰がテストするのか、誰がレビューするのか、何を確認してからリリースするのか。何も決まっていない状態で「とりあえず動くものを作る」を繰り返していた。

## 最初の対策: エージェントの責務を分割する

当初のエージェント構成は4体だった。

```
dev_manager（管理）
  → programmer（実装 + テストコード作成）
  → qa_agent（レビュー + テスト実行 + 品質判定）
  → ux_agent（UX検証）
```

2つの問題があった。

1. **programmerが自分のコードのテストを書く** — 自分が書いたコードに対するテストは甘くなりやすい。人間でもAIでも同じだ
2. **qa_agentの責務が多すぎる** — テスト設計・テスト実行・コードレビュー・品質判定を1体で担当。どれも中途半端になる

そこで6体に分割した。

```
dev_manager（管理）
  → programmer（実装のみ）
  → qa（テスト設計 + レビュー + 品質判定）
  → tester（テストコード作成 + テスト実行）← 新設
  → ux_agent（UX検証）
  → env_manager（環境構築・変更の一元管理）← 新設
```

**「テストを書く人」と「コードを書く人」を分けた。** これで、programmerが見落とすエッジケースをtesterが拾うようになった。

## しかし、工程スキップが発生する

6体に分けても新しい問題が出た。**エージェントが開発工程をスキップする**のだ。

本来の工程はこうだ:
```
計画 → 承認 → 実装 → レビュー → テスト → 品質判定 → 完了
```

しかし実際に起きたのはこうだった:
```
ユーザー「この機能を追加して」
→ 指示を受けたエージェントが、programmerに渡さず直接ソースコードを編集
→ テストなし、レビューなし、品質判定なしで変更が完了してしまう
```

これが7件連続で発生した（INC-027〜033）。**管理役のエージェントが「工程を通すより自分でやった方が早い」と判断して、直接コードを触ってしまう。** 人間の組織でも管理職が現場の仕事を巻き取ってしまう問題は起きるが、まさか同じことがAIでも起きるとは思わなかった。

## 第二の対策: hookで物理的にブロックする

「CLAUDE.mdに『工程を守れ』と書いても守らない」ことがわかった。プロンプトの改善では解決しない。

ここで使ったのが、Claude Codeの**hook機能**だ。hookとは、ファイル編集やコマンド実行などのアクションの前後に、任意のシェルスクリプトを自動実行する仕組みだ。スクリプトがエラーを返すと、そのアクションがブロックされる。

これを使って「工程ゲート」を作った。

```bash
# check_process_gate.sh（概要）
# src/ や test/ を編集しようとしたとき、
# 計画と承認が完了していなければ編集をブロック

if [[ ! -f "$PROJECT_ROOT/data/process/current_plan.md" ]]; then
  echo "BLOCK: 計画ファイルがありません。先にdev_managerで計画を作成してください"
  exit 1
fi

if [[ ! -f "$PROJECT_ROOT/data/process/plan_approved.json" ]]; then
  echo "BLOCK: 計画が承認されていません"
  exit 1
fi
```

**「お願いだから守って」ではなく、物理的にブロックする。** これが効いた。計画なしにコードを書こうとすると、エディタの操作自体が拒否される。

さらに、全体で12工程のチェックリストを作った:

1. 障害台帳に記録（障害起因の場合）
2. dev_managerが計画を策定
3. ユーザーが計画を承認
4. 調査（ドメイン知識の獲得）
5. programmerが実装
6. qaがコードレビュー
7. testerがテスト実行
8. dev_managerが品質ゲート判定
9. ux_agentとユーザーがUAT（受入テスト）
10. 障害台帳を更新
11. biz_mgrがリリース判定
12. デプロイ

個人開発にしては重い。でも、**工程を省略した結果として100件の障害が起きたことを考えると、これが最小限のプロセスだった。**

## 学んだ設計原則

ここまでの試行錯誤を振り返ると、4つの原則に集約できる。

これらを整理しているときに、Anthropicが公開した[Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)という記事に出会った。「ハーネスエンジニアリング」——AIエージェントの動作を安定させるために、プロンプトではなく外部の仕組み（ツール制限、チェックポイント、ガードレール等）を設計する考え方だ。

読んで驚いた。自分がこの1週間、手なりでやっていたこと——hookで工程をブロックする、障害台帳を蓄積する、情報を階層化する——がまさにハーネスエンジニアリングそのものだった。概念として名前がつくと、それまで場当たり的だった対処が「設計」に変わる。以降、この考え方を意識的に徹底するようになった。

### 原則1: プロンプトを直すな、環境を直せ

エージェントが間違いを繰り返すとき、「もっと注意して」とプロンプトに書きたくなる。でもそれは効かない。

代わりに、**「間違いが起きない環境」を作る**。

| 繰り返された失敗 | プロンプト改善（効かない） | 環境側の対処（効いた） |
|-----------------|-------------------------|---------------------|
| 工程をスキップする | 「必ず工程を守ること」と追記 | hookで物理的にブロック |
| セレクタを推測で書く | 「実サイトを確認すること」と追記 | チェックリストに「実DOM確認」を必須項目として追加 |
| テストを書かない | 「テストを書くこと」と追記 | tester専任エージェントを新設 |

**同じ失敗が繰り返されるなら、それはエージェントの問題ではなく環境設計の問題。**

### 原則2: 責務を分けろ、でも分けすぎるな

4体 → 6体の分割は効果があった。しかし闇雲に増やすとエージェント間の情報伝達コストが増える。

判断基準はシンプルだ:
- **分けるべき**: 利益相反がある（実装者がテストを書く等）
- **分けるべきでない**: 情報伝達のコストが分離の利益を上回る

6体を超えたあたりから「Aに伝えたことがBに伝わっていない」問題が目立ち始めた。

### 原則3: 情報は階層化して管理する

エージェントのコンテキストウィンドウ（一度に読める情報量）には限りがある。プロジェクトのルール・規約・ドメイン知識をすべて1ファイルに書くと、肝心な部分が埋もれる。

そこで、CLAUDE.md（エージェントが最初に読むファイル）は目次に徹し、詳細は個別ファイルに分離した。

```
CLAUDE.md（~100行の目次・ポインタ集）
  ├→ knowledge/*.md（ドメイン知識）
  ├→ doc/00_process/*.md（プロセス定義）
  └→ doc/adr/*.md（過去の設計判断記録）
```

エージェントは必要な情報だけを読みに行く。全部を一度に読ませる必要はない。

### 原則4: 障害台帳が最高のドキュメントになる

100件超の障害台帳は、意図せず最も価値のあるドキュメントになった。

各エントリにこの6項目を記録している:
- **現象**: 何が起きたか
- **原因**: 直接の原因
- **原因の水平展開**: 同じパターンが他にないか
- **根本原因**: なぜその原因が生まれたか（5 Whys）
- **根本原因の水平展開**: 根本原因レベルで他に影響がないか
- **対応内容**: 何をどう修正したか

**「なぜそうなったか」「なぜ見逃したか」の記録が蓄積されると、qaエージェントがレビュー時に過去の障害パターンを参照するようになった。** たとえば「このコードは INC-003（二重処理）と同じパターンのリスクがある」といった指摘が出るようになる。

## その先: 開発チームから事業判断チームへ

開発プロセスが安定してきたので、さらに上位の層を追加してみた。

```
事業戦略層（entrepreneurエージェント）
  仮説立案 → 実行指示 → KPI検証 → ピボット/改善

業務管理層（biz_mgrエージェント）
  計画 → 実行管理 → 進捗確認 → 調整

開発層（dev_managerエージェント）← ここまでが前述の話
  設計 → 実装 → レビュー/テスト → 品質判定
```

事業仮説の検証をentrepreneurエージェントに、実行管理をbiz_mgrエージェントに委譲した。entrepreneurは仮説を立て、financial_managerがコストを分析し、biz_mgrが開発チームに実行指示を出す。

この仕組みで別のSaaSの企画を検証してみたところ、**「このサービスは作るべきではない」という判断が出た。** ターゲットにリーチするチャネルがない、0円で代替可能、提供するコンテンツとユーザーのニーズにギャップがある。1日で5つの仮説を検証し、撤退判断に至った。

コードを1行も書く前に、作るべきでないものを作らずに済んだ。これは「動くコードを書く」よりもずっと価値のある成果だった。

## まとめ

| 学び | 詳細 |
|------|------|
| エージェントは指示を守らない | プロンプトの改善よりも、物理的な制約（hook）が有効 |
| 責務の分離は人間と同じ | 実装者とテスターを分けると品質が上がる |
| 障害台帳は資産 | 「なぜ失敗したか」の蓄積がエージェントのレビュー精度を改善する |
| 情報は階層化する | コンテキストウィンドウは有限。目次→詳細の構造を作る |
| プロセスの重さは障害の数で正当化される | 100件の障害が12工程チェックリストを生んだ |

**「AIに仕事を任せる」のは「AIにプロンプトを渡す」ことではない。AIが正しく動ける環境を設計することだ。**
