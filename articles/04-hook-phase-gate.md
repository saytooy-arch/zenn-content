# AIエージェントの工程遷移をhookで物理的に制御する

## TL;DR

Claude Codeのマルチエージェント開発で「ルールを書いても守られない」問題に直面した。解決策は**hookスクリプトで物理的にブロック**すること。全7工程（ED→SD→MF→UT→IT→ST→UAT）にゲートを設け、前工程の完了判定チェックリストにレビュー者署名がなければ次工程のファイル編集をexit 2でブロックする仕組みを構築した。

## 背景：ルールを書いても守られない

Claude Codeでマルチエージェント開発チーム（15+エージェント）を組んでSaaS開発をした。quality_process.mdに「工程スキップ禁止」と書いたのに、実際には：

- 画面設計書なしで実装が始まった
- テスト仕様書なしでテストが書かれた
- レビュー記録なしでデプロイしようとした
- 18件のインシデントが発生

AIエージェントも人間と同じで、ルール文書を読んでいても実行時に省略する。

## 解決策：hookスクリプトで物理的にブロック

Claude CodeのPreToolUse hookを使い、ファイル編集前に前工程の完了を自動チェックする。

### アーキテクチャ

```
ユーザー/エージェント → Edit/Write ツール呼び出し
  → PreToolUse hook (check_process_gate.sh)
    → current_phase.txt から現在工程を取得
    → 工程マッピングで必要な前工程ゲートを特定
    → ゲートファイルの署名を確認
    → 署名なし or FAIL → exit 2 (ブロック)
    → 署名あり & PASS → exit 0 (許可)
```

## チェックリスト設計の落とし穴

### ダブルチェックは形骸化する

最初は「確認者✅ / レビュー者✅」の2列を設けた。しかし、レビュー者は確認者のチェック済みを見て「もう確認されてるからOK」と形骸化する。

### 正解：確認者は参照のみ、レビュー者がチェック

- **確認者**：チェックリストを読んで作業内容を把握し、作業する。チェック欄は持たない。
- **レビュー者**：作業結果を検証し、チェック欄に記入する。

チェック欄を持つのはレビュー者のみ。

## hookをすり抜ける方法と対策

hookで全工程を制御しても、エージェントが `Bash(claude -p "...")` で別プロセスを起動すると、hookが適用されない。

### 問題

```
# これはhookを通る（ブロックされる）
Agent tool → subagent_type: programmer

# これはhookを通らない（すり抜ける）
Bash(claude -p "あなたはprogrammerです...")
```

### 対策

`check_bash_command.sh` で `claude -p` 自体をブロック。

### PreToolUse hookの適用範囲

重要な注意: PreToolUse hookはClaude Code**内**のツール呼び出し時のみ発火する。cronジョブ等から直接 `claude -p` を呼ぶ場合は影響しない。

## L3→L1昇格パターン：定義記載のみからhook強制へ

品質プロセスのルールには3つの強制レベルがある:

| レベル | 手段 | 違反時 | 確実性 |
|---|---|---|---|
| **L1: hook** | check_process_gate.sh (exit 2) | ファイル編集ブロック | 確実 |
| **L2: ツール権限** | Agent定義のtools制限 | 操作自体が不可能 | 確実 |
| **L3: 定義記載** | agent定義・CLAUDE.mdに記載 | 違反しても動作する | 不確実 |

問題は、重要なルールが「L3（記載のみ）」に留まっていること。体系的に分析すると6件の穴が見つかった:

- dev-managerが各工程完了を確認していない → **L1昇格**: `REQUIRE_DEV_MANAGER_GATE`
- test-architectレビューなしでテスト実装可能 → **L1昇格**: `REQUIRE_TEST_DESIGN_REVIEW`
- incident_ledger.mdを誰でも編集可能 → **L1昇格**: `REQUIRE_INCIDENT_COMMANDER`
- 確認者とレビュー者が同一人物でも通過 → **L1昇格**: `validate_gate_signatures()`

### 委譲マーカーパターン：「誰が」を間接制御する

hookでは「誰が編集しているか」を直接判別できない。しかし、**前提条件ファイルの有無**で間接的にアクセス制御できる。

```
incident-commander 起動
  → data/process/incident_commander_delegation.json 作成
  → incident_ledger.md 編集可能に
  → 対応完了時に delegation.json 削除
  → 他のエージェントは delegation.json を作成する手段がない
    → incident_ledger.md 編集不可
```

これを**委譲マーカーパターン**と呼ぶ。特定のエージェントのみが作成・削除する権限を持つJSONファイルの存在で、ファイル編集の可否を制御する。

## まとめ

- ルール文書だけでは守られない。hookで物理的にブロックして初めて「強制」
- 全工程にゲートを置くことで、品質が構造的に担保される
- チェックリストは「レビュー者のみがチェック」が正しい設計
- hookにも抜け道がある。`claude -p` による別プロセス起動を塞ぐことで完全性が向上する
- 重要なルールが「L3（記載のみ）」に留まっていないか定期的に監査し、L1に昇格させる
- hookで「誰が」を判別できなくても、委譲マーカーパターンで間接制御できる
