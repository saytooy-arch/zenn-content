## はじめに

飲食店向け原価計算ツールの開発中、ドメインモデルと実装を照合したところ5件の設計欠陥が見つかった。

- ml→g変換が水の密度（1.0）固定。油なら40%ずれる
- 「個」「本」単位の食材は栄養計算を完全スキップ
- 食材単価0円が許容されている（業務上ありえない）

なぜこれらを設計段階で防げなかったのか。5段階の深掘りで到達した答えは「入力仕様で『等』を禁止する」という、驚くほどシンプルなルールだった。

## 5件の設計欠陥

ドメインモデル（業務知識から構築、コードからのリバースではない）と実装を照合して検出した5件:

| ID | 種別 | 内容 |
|----|------|------|
| G1 | 命名不一致 | ドメイン「レシピ」に対し実装は「menu」 |
| G2 | ドメイン制約漏れ | 食材単価0円が許容（業務上ありえない） |
| P1 | 精度問題 | ml→g変換が水の密度（1.0）固定。油（0.9）で10%、蜂蜜（1.4）で40%ずれる |
| P2 | 機能欠損 | 「個」「本」単位は栄養計算を完全スキップ |
| P3 | UX問題 | 変換不可時のreason表示が誤り |

## 深掘り: なぜ設計できなかったか

### Level 1: ドメインモデリング工程がなかった

G1（命名不一致）とG2（単価0円許容）は、ドメインモデリング工程があれば防げた。業務概念を整理すれば「レシピ」という用語や「単価は正の数」という制約は自然に出てくる。

**しかし、P1とP2はドメインモデリングだけでは防げない。**

ドメインモデルに「単位変換」という概念を入れても、「mlとgの変換には密度が必要」「密度は食材ごとに異なる」という知識は、ドメインモデリングという工程からは導出されない。

### Level 2: ドメインモデルは「知っていることを整理する」工程

ここで重要な認識に至った:

> ドメインモデリングは「知っていることを整理する」工程であり、「知らないことを発見する」工程ではない。

開発者がドメイン知識を十分に持っていない場合、ドメインモデルをいくら丁寧に書いても、知らない制約は出てこない。

### Level 3: 業務データの調査プロセスがなかった

P1を防ぐには、食品成分表の公式ドキュメントを読み、「密度」という概念の存在を知る必要があった。つまり**外部文書を調査するプロセス**が必要。

しかし「外部文書を読め」というルールだけでは、**何を調べればいいかわからない**。

### Level 4: 「何を調べるか」がわからない

「単位変換を設計せよ」という指示を受けたとき、「密度について調べよう」という発想が出るのは、密度の存在を既に知っている人だけだ。

では、知らない人はどうすれば必要な調査項目を発見できるのか。

### Level 5: 入力仕様で「等」を禁止する

答えはこうだ:

> 入力仕様で「等」「その他」「など」を禁止し、**取りうる値を全件列挙**させる。

例えば単位の入力仕様を書くとき:

```
# NG: 曖昧な仕様
単位: g, ml, 個 等

# OK: 全値列挙
単位: g, kg, mg, ml, L, cc, 個, 本, 枚, 束, パック, 袋, 缶, 丁, 合
```

全値を列挙すると、**各値に対する処理を考えざるを得ない**:

- 「gとkgの変換は？」→ 単純な倍率変換
- 「mlとgの変換は？」→ **密度が必要** → 密度はどこから取得する？ → 食品成分表を調べる必要がある
- 「個とgの変換は？」→ **個あたり重量が必要** → どこから取得する？ → 食品成分表の廃棄率・重量変化率を調べる必要がある

つまり「等」を禁止して全値列挙するだけで、**必要なデータ要件が自動的に導出される**。

## なぜ「等」が危険なのか

「等」は3つの問題を引き起こす:

1. **列挙の省略を正当化する**: 「g, ml, 個 等」と書いた瞬間、残りの単位の処理を考えなくてよくなる
2. **未知の値を隠蔽する**: 「等」の中に何が含まれるか誰もわからない。レビューでも検出できない
3. **テストケースが不完全になる**: 列挙されていない値はテスト対象にならない

## 実装: ED工程でのルール化

このルールをプロセスに組み込んだ方法:

### 1. 入力仕様テンプレートの改訂

ED（詳細設計）工程の入力仕様セクションで、「等」「その他」「など」「etc」を禁止。全入力フィールドについて取りうる値を全件列挙する。

### 2. /loop自問検証

対策設計後に「本当にこれで解決する？」と4観点で自問する検証ループを導入:

1. この対策でG1〜G2（ドメイン制約系）は防げるか？
2. この対策でP1〜P2（データ要件系）は防げるか？
3. この対策が機能しない条件は？
4. より根本的な対策はないか？

3回の自問ループで「ドメインモデリング追加」→「外部文書調査」→「入力の全値列挙」に到達した。

## 適用範囲と限界

### 有効なケース

- 列挙可能な入力値（単位、ステータス、カテゴリ等...ではなく、具体的に列挙する）
- 型定義でunion型にできるもの
- マスタデータとして管理されるもの

### 限界

- 自由入力テキスト（食材名など）には適用できない
- 外部APIのレスポンスなど、自分でコントロールできない値
- 将来追加される可能性が高い値（ただしこの場合も「現時点の全値」を列挙し、拡張ポイントを明示する方が安全）

## まとめ

設計品質の問題を深掘りした結果:

1. ドメインモデリング → 「知っていること」は整理できるが「知らないこと」は発見できない
2. 外部文書調査 → 「何を調べるか」がわからなければ読めない
3. **入力の全値列挙** → 各値の処理を考えることで、必要な調査項目が自動的に導出される

「等」を禁止する。たったこれだけで、設計段階で「何を調べるべきか」が見えるようになる。
