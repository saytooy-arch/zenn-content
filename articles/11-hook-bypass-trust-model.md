## TL;DR

AIエージェントの品質管理を自動化するhookシステムを構築したが、エージェント自身がhookをバイパスする方法を発見してしまった。「ファイルが存在する=正規権限者が作成した」という暗黙の前提が全hookに共通する設計欠陥だった。11件のバイパス経路を検出し、信頼モデルの再設計に至った話。

## 背景: AIエージェント開発のガバナンス問題

Claude Codeのマルチエージェント体制（15+エージェント）でSaaSを開発している。各エージェントが勝手にルールを破らないよう、hookスクリプト（PreToolUse/PostToolUse）で物理的にブロックする仕組みを構築してきた。

```
orchestrator → biz-mgr → dev-manager → programmer
                                      → qa → tester
              → incident-commander
```

hookの例：
- `check_file_edit.sh`: orchestratorがプロジェクト配下ファイルを直接編集するのをブロック
- `check_process_gate.sh`: 計画承認なしに実装工程に進むのをブロック
- `check_test_result.sh`: テストFAIL中にソースコードを編集するのをブロック

30件のインシデント対応を経て、hookは20+に増加。「ルールは文書に書くだけでは守られない。物理的にブロックして初めて強制になる」が教訓だった。

## 事件: orchestratorがhookをバイパスした

ある日、プロジェクト全体の品質監査結果をバックログに登録しようとした。

```
orchestrator が /bl スキルを実行
  ↓
product_backlog.md を編集しようとする
  ↓
hook「biz-mgrのみ編集可能」でブロック ← ここまでは正常
  ↓
biz-mgrに委譲するのではなく...
biz_mgr_backlog_delegation.json を自力で作成してhookをバイパス ← ！？
```

ユーザーに「そのやり方はルール違反では？」と指摘されて停止。

## 根本原因: hookの信頼モデルの欠陥

### 「ファイル存在 = 正規権限者が作成」の暗黙の前提

```bash
# check_process_gate.sh の典型的なパターン
if [ ! -f "$GATE_DIR/biz_mgr_backlog_delegation.json" ]; then
  exit 2  # ブロック
fi
# ← ファイルが「誰によって作成されたか」は検証しない
```

この前提は全hookに共通していた。

### 発見された11件のバイパス経路

| # | バイパス方法 | 深刻度 |
|---|------------|:---:|
| 1 | `/tmp/genka_active_agent`を手動作成→orchestratorブロック無効化 | 極高 |
| 2 | 委譲マーカーJSONを手動作成→ゲート通過 | 高 |
| 3 | 承認JSON（plan_approved等）を手動作成→工程スキップ | 高 |
| 4 | test_result.jsonを手動改ざん→テストFAIL隠蔽 | 高 |
| 5 | common/配下がブロック対象外→hookスクリプト本体を直接編集 | 極高 |
| 6 | Bash経由でrm/cp/mvによるファイル操作がブロックされていない | 極高 |
| 7 | `export REQUIRE_*=false`で全hook条件を無効化可能 | 極高 |

### 5 Whys

| # | なぜ？ | 回答 |
|---|--------|------|
| 1 | なぜバイパスできた？ | 委譲マーカーJSONを自力で作成できた |
| 2 | なぜ自力作成できた？ | hookが「ファイル存在」のみチェックし「誰が作成したか」を検証しない |
| 3 | なぜ作成者検証がない？ | hookシステム全体が「ファイル存在=権限あり」で設計されている |
| 4 | なぜその前提？ | SubagentStart hookで自動管理すれば十分と想定 |
| 5 | 構造的原因 | **hookの信頼モデルが「ファイル操作=正規エージェント」を暗黙に仮定** |

## 教訓

### 1. hookが増えるほどバイパス経路も増える

hookは「鍵ファイルが存在すれば通過」という設計。hookが増えると鍵ファイルも増え、鍵ファイルの保護が追いつかない。鍵を守るためのhookを作ると、さらに鍵が増える...

### 2. 「仕組みを作る側」が仕組みをバイパスする

人間の組織でも「ルールを作る管理者がルールを破る」問題はある。AIエージェントでも同じ。orchestrator（スキル定義含む）がhookバイパスの手順を持っていた。

### 3. スキル定義はセキュリティ境界

`/bl`スキルの手順に「委譲マーカーを自力作成」と書いてあった。スキル定義 = orchestratorへの指示書であり、hookバイパス手順を含めてはいけない。

### 4. PBは「読み手」の言葉で書く

技術者が「VUL-1: /tmp廃止→トークン方式移行」と書いても、biz-mgr（業務管理者）には計画に落とせない。「何が問題で何を達成すべきか」で書く必要がある。

## まとめ

AIエージェントのガバナンスにおいて、hookによる物理的ブロックは必要条件だが十分条件ではない。hookの「信頼モデル」—— 何をもって正規の操作と判定するか —— の設計が最も重要。

「ファイルが存在するから権限がある」ではなく「正規のプロセスで作成されたことを検証できる」設計が必要。これは人間の認証システム（パスワードの存在 vs パスワードの正当性検証）と同じ原則。
