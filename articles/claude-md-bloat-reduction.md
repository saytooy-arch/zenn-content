## TL;DR

Claude Codeのマルチエージェント構成で開発していたら、CLAUDE.mdが339行まで膨張した。「CLAUDE.mdの読者はOrchestratorである」という原則に立ち返り、下位エージェントが自分の定義から参照できる情報を全削除。4プロジェクト合計で977行→494行（49%削減）に圧縮した。

## 前提: Claude Codeのマルチエージェント構成

Claude Codeには、`.claude/agents/` にMarkdownファイルを置くことでエージェントを定義する機能がある。各エージェントに使えるツールやモデルを制限でき、Agent toolで呼び出すと独立したサブプロセスとして動作する。

この機能を使って、15+体のエージェントによる開発チームを組んでいる。ユーザーの指示を受けるOrchestrator（トップレベルエージェント）が、dev-manager、programmer、qa、tester等の専門エージェントにタスクを委譲する構成だ。

**CLAUDE.md**はプロジェクトのルートに置く設定ファイルで、Claude Codeが作業を開始するたびに自動的に読み込まれる。プロジェクトの概要、規約、制約などを記述する。

## 問題: CLAUDE.mdが際限なく太る

マルチエージェント体制を運用していると、CLAUDE.mdに書きたい情報が増え続ける。

- 委譲先エージェント一覧・組織構造図
- 開発フロー（計画→実装→テスト→品質ゲート）
- 品質ゲート基準（8項目）
- 障害管理フロー
- ドキュメント構成（11階層のフォルダツリー）
- 共通リソースカタログ（ナレッジ6件、テンプレート12件、スクリプト11件）
- イテレーション管理、レビュー記録形式、ADR、セッション管理...

結果、最も複雑なプロジェクト（飲食店向けSaaS）のCLAUDE.mdは**339行**に達した。

### なぜ問題か

CLAUDE.mdはプロジェクトディレクトリで作業するたびに**自動ロード**される。339行のコンテキストが毎回消費される。しかも、その大半はOrchestratorが直接使わない情報だった。

## 気づき:「読者は誰か」

CLAUDE.mdを読むのは誰か？ → **Orchestrator**（トップレベルエージェント）。

Orchestratorの仕事は「適切なエージェントに委譲すること」。Orchestratorが必要な情報は:

1. **誰に委譲するか**（委譲先エージェント一覧）
2. **いつ誰を呼ぶか**（判断基準・ルーティング）
3. **制約**（やってはいけないこと）
4. **プロジェクト概要**（ルーティング判断の文脈）

逆に、Orchestratorが**不要**な情報:

- 品質ゲート基準 → qa / dev-managerが使う
- 開発フロー詳細ステップ → dev-managerが管轄
- ドキュメント構成ツリー → dev-managerが委譲時に指示
- 共通リソースカタログ → 各エージェントが自身の定義から参照
- レビュー記録形式、ADRテンプレート → 各担当が知っている

**CLAUDE.mdに書かれている情報の大半は「Orchestrator以外が使う情報」だった。**

## 前提: 3層エージェント定義

この削減が可能なのは、エージェント定義が3層構造になっているからだ。

```
第1層: ネイティブ定義（.claude/agents/*.md）
  → エージェントのペルソナ + 共通テンプレート参照

第2層: 共通テンプレート（common/templates/agents/*.md）
  → 役割定義 + 共通ナレッジ参照

第3層: プロジェクト固有定義（<project>/agents/*.md）
  → プロジェクト固有のカスタマイズ
```

例えば `qa` エージェントは:
- 自身の定義で `common/knowledge/quality_process.md` を読むよう指示されている
- テスト設計テンプレートの存在を知っている
- 品質ゲート基準は共通ナレッジから取得する

つまり、CLAUDE.mdにこれらのカタログを載せなくても、qaエージェントは自分の仕事に必要な情報を知っている。

```bash
# 検証: エージェント定義内の共通ナレッジ参照を検索
$ grep -r "common/knowledge\|common/templates" .claude/agents/
# → 15エージェントが明示的に共通ナレッジを参照していた
```

## 削減の実施

### 削除した内容

1. **配下エージェント詳細テーブル** → 組織構造図（ツリー表記）に集約済み
2. **品質ゲート基準テーブル** → 品質設計ナレッジに完全版あり（qa / dev-managerが参照）
3. **開発フロー詳細ステップ** → 品質プロセスナレッジに完全版あり（dev-managerが参照）
4. **ドキュメント構成ツリー** → ドキュメント構造定義に完全版あり（dev-managerが参照）
5. **共通リソースカタログ** → 各エージェント定義から直接参照
6. **工程ゲートスクリプト一覧** → hookが自動実行 + エージェント定義に記載

### 残した内容

- 委譲先エージェント一覧（Orchestratorの判断材料）
- 組織構造図（ルーティングの全体像）
- 障害/要望時のルーティングフロー
- セッション管理（Orchestrator自身が実行）
- Key Constraints（Orchestratorへの制約）
- Project Overview（ルーティング判断の文脈）

## さらに踏み込む: ドキュメント4分類

[nyosegawa.com の記事](https://nyosegawa.com/posts/docs-in-agent-era/)で提唱されている4分類を適用すると、さらに削減できた。

| 分類 | 定義 | 例 | 対応 |
|------|------|-----|------|
| 導出可能 | コードから再構成可能 | File Structure、API仕様 | **書かない** |
| 検証可能 | 機械的にtrue/false判定可能 | 「工程スキップ禁止」 | **hookに移す** |
| 不変の記録 | ある時点の決定理由 | ADR、ポストモーテム | **Append-only保持** |
| 還元不能 | コードに落とせない知識 | Why/Why not、法規制 | **自然言語で維持** |

### File Structureは書かない（導出可能）

ETH ZurichとLogicStar.aiの[研究](https://arxiv.org/abs/2602.11988v1)で、「ディレクトリ構造の概要をCLAUDE.mdに書いてもAgentのファイル発見速度は向上しなかった」という結果が出ている。

全CLAUDE.mdからFile Structureセクション（計70行）を削除した。Agentは`glob`や`ls`で直接ファイルを探せる。

### hookで強制済みの制約は書かない（検証可能）

ここが一番の発見だった。

Claude Codeの**hook**は、ファイル編集やコマンド実行などのアクションの前後に、任意のシェルスクリプトを自動実行する仕組みだ。スクリプトが`exit 2`を返すとそのアクションがブロックされ、`stderr`の内容がエージェントに表示される。

つまり、hookがブロックした時点で「なぜブロックされたか」「何をすべきか」がエージェントに自動伝達される。CLAUDE.mdに書く必要がない。

```bash
# hookスクリプトの例: 計画なしの実装をブロック
echo "[PROCESS GATE] ソースファイルの編集にはまず計画が必要です。" >&2
echo "  計画ファイルが存在しません。" >&2
echo "  dev-manager に計画策定を依頼してください。" >&2
exit 2  # ブロック + stderrをエージェントに表示
```

Key Constraintsから4項目を削除できた:
- ~~Windows側操作は原則禁止~~ → hookでブロック済み
- ~~工程スキップ禁止~~ → hookでブロック済み
- ~~src/test 編集前に計画・承認が必要~~ → hookでブロック済み
- ~~コーディング規約は共通ナレッジに従う~~ → hookでブロック済み

残したのはhookで強制できない「還元不能」な制約のみ:
- 公開情報のみ使用（社内情報禁止）
- Web完結型サービス

**hookのエラーメッセージがドキュメントを代替する。** 検証可能な制約は、CLAUDE.mdに「やるな」と書く代わりにhookで物理的にブロックすれば、ドキュメントとしても不要になる。

## ドキュメントROIの計測

削減だけでなく「実際に使われているか」を計測する仕組みも作った。

Claude Codeのセッションログ（`~/.claude/projects/*.jsonl`）には全ツール呼び出しが記録されている。これを解析して:

1. **参照頻度ランキング** — どのドキュメントが何回Readされたか
2. **未参照ドキュメント検出** — 一度も読まれていないドキュメント

```bash
$ bash docs_auditor.sh --days 30

# 参照頻度ランキング
| # | パス                                | 参照回数 |
| 1 | doc/incident_ledger.md              | 19       |
| 2 | .claude/agents/dev-manager.md       | 16       |
| 3 | common/knowledge/quality_process.md | 12       |
...

# 未参照ドキュメント
| common/knowledge/competitive_analysis_guide.md | 未参照 |
| common/templates/domain_knowledge_checklist.md  | 未参照 |
```

「書いたけど誰も読んでいない」ドキュメントを定量的に検出できる。

## 肥大化防止の仕組み

削減は1回きりの作業だが、再膨張を防ぐ仕組みが必要だ。

1. **CLAUDE.md行数チェック hook** — 編集後に100行超で警告
2. **パス参照検証スクリプト** — CLAUDE.md内の壊れたパスを検出
3. **docs-auditor** — 定期的にROI計測し、未参照ドキュメントを特定

## 最終結果

| プロジェクト | 最初 | 最終 | 削減 |
|------------|------|------|------|
| 飲食店向けSaaS | 339行 | 86行 | **-75%** |
| キュレーションサービス | 159行 | 70行 | -56% |
| ニュース配信パイプライン | 210行 | 107行 | -49% |
| レシピアプリ | 201行 | 163行 | -19% |
| **合計** | **977行** | **494行** | **-49%** |

最も複雑なプロジェクトほど削減率が高い。339行→86行（75%削減）。逆に、レシピアプリは元々シンプルでエージェント数も少ないため、削減余地が小さかった。

## 教訓

1. **CLAUDE.mdの読者を意識する** — Orchestratorが使わない情報は載せない
2. **エージェント定義の3層構造が前提** — 各エージェントが自分の仕事に必要な情報を自身の定義から取得できる設計が先
3. **「圧縮」ではなく「削除」** — テーブルをコンパクトリストにする前に、「そもそも必要か？」を問う
4. **hookのエラーメッセージがドキュメントを代替する** — 検証可能な制約はhookに移せば、ドキュメントに書く必要がなくなる
5. **計測なき最適化は推測に過ぎない** — 参照頻度を計測し、データに基づいて削減する
